FASE 1: Mejoras Criticas (Semanas 1-4)
1. Implementacion de MFA (Autenticacion Multi-Factor) Esfuerzo:

Objetivo: Agregar segundo factor de autenticacion (TOTP) para todos los usuarios, obligatorio para administradores.

Aplicaciones Afectadas

Parse (Primero) Hub Checkpoint Rendiciones AxiomaDocs
Diseno

Arquitectura MFA centralizada en Parse, SDK para apps cliente

3 dias
Desarrollo

Backend Parse + migracion DB + endpoints + UI en todas las apps

12 dias
Pruebas

Testing unitario, integracion, UAT con usuarios reales

4 dias
Despliegue

Rollout gradual: admins primero, luego todos

3 dias
Tareas de Desarrollo
  • Parse Backend: Instalar speakeasy/otplib para generacion TOTP
  • Parse Backend: Crear modelo MfaSecret (userId, secret, enabled, backupCodes)
  • Parse Backend: Endpoint POST /auth/mfa/setup - Generar secreto y QR
  • Parse Backend: Endpoint POST /auth/mfa/verify - Validar codigo TOTP
  • Parse Backend: Endpoint POST /auth/mfa/disable - Desactivar (requiere codigo)
  • Parse Backend: Modificar POST /auth/login - Agregar paso MFA si habilitado
  • Parse Backend: Generar backup codes (10 codigos de un solo uso)
  • Parse Frontend: Pantalla de configuracion MFA con QR
  • Parse Frontend: Modal de verificacion MFA en login
  • Parse Frontend: Pantalla de backup codes
  • Hub/Checkpoint/Rendiciones/Docs: Integrar flujo MFA en login
  • Migracion DB: Agregar tabla mfa_secrets y campo mfaEnabled en users
  • Admin: Forzar MFA para roles SUPER_ADMIN y ADMIN
Plan de Pruebas
  • Unit tests: Generacion de secretos, validacion TOTP, backup codes
  • Integration tests: Flujo completo login -> MFA -> dashboard
  • Test de tiempo: Verificar ventana de 30 segundos del TOTP
  • Test de backup codes: Uso unico y regeneracion
  • Test de fuerza bruta: Rate limiting en verificacion MFA
  • UAT: 3 usuarios admin prueban el flujo completo
  • Test de recuperacion: Proceso cuando usuario pierde dispositivo
Dependencias

Ninguna. Esta mejora puede comenzar inmediatamente.

Riesgos y Mitigacion
  • Usuarios sin smartphone: Proveer backup codes impresos
  • Perdida de dispositivo: Proceso de recuperacion via email + validacion manual
  • Resistencia al cambio: Rollout gradual, comunicacion previa
2. Plan de Respuesta a Incidentes (IRP) Esfuerzo:

Objetivo: Documentar procedimientos de respuesta ante incidentes de seguridad con roles, escalamiento y comunicacion.

Aplicaciones Afectadas

Todas (Documentacion) Digital Ocean
Investigacion

Revisar templates NIST, adaptar a contexto Syncro

2 dias
Documentacion

Redactar IRP completo con procedimientos DO

5 dias
Revision

Validar con equipo tecnico y stakeholders

2 dias
Tabletop

Simulacro de mesa con escenario ficticio

1 dia
Contenido del IRP
  • Seccion 1: Definicion de incidente y niveles de severidad (Bajo/Medio/Alto/Critico)
  • Seccion 2: Equipo de respuesta - Roles y contactos (Lider, Tecnico, Comunicacion, Legal)
  • Seccion 3: Procedimiento de deteccion - Como identificar un incidente
  • Seccion 4: Procedimiento de contencion inmediata (aislamiento de Droplets, revocacion de tokens)
  • Seccion 5: Procedimientos Digital Ocean (snapshots de emergencia, logs, firewall rules)
  • Seccion 6: Procedimiento de erradicacion y recuperacion
  • Seccion 7: Comunicacion interna y externa (templates de notificacion)
  • Seccion 8: Post-mortem y lecciones aprendidas
  • Anexo A: Checklist de respuesta rapida (1 pagina)
  • Anexo B: Contactos de emergencia (DO Support, ISP, Legal)
  • Anexo C: Comandos utiles Digital Ocean CLI
Validacion
  • Revision tecnica por desarrollador senior
  • Revision por responsable de operaciones
  • Tabletop exercise: Escenario "Breach de credenciales de usuario admin"
  • Documentar tiempos de respuesta del ejercicio
  • Ajustar IRP segun hallazgos del tabletop
Dependencias

Ninguna. Puede ejecutarse en paralelo con MFA.

FASE 2: Mejoras de Prioridad Alta (Semanas 2-12)
3. Escaneo de Vulnerabilidades en CI/CD Esfuerzo:

Objetivo: Automatizar escaneo de dependencias vulnerables en cada push/PR usando GitHub Actions + Snyk o Dependabot.

Aplicaciones Afectadas

Parse Hub Checkpoint Rendiciones AxiomaDocs
Configuracion

Setup Snyk/Dependabot en cada repositorio

2 dias
Integracion

GitHub Actions workflows, branch protection

3 dias
Remediacion Inicial

Resolver vulnerabilidades existentes criticas

3 dias
Documentacion

Proceso de triaje y actualizacion

1 dia
Tareas de Implementacion
  • Crear cuenta Snyk (free tier para proyectos open source/small teams)
  • Conectar repositorios de GitHub a Snyk
  • Configurar .snyk en cada proyecto con politicas de severidad
  • Crear workflow .github/workflows/security-scan.yml
  • Configurar branch protection: bloquear merge si hay vulnerabilidades criticas
  • Habilitar Dependabot alerts en GitHub como backup
  • Ejecutar primer escaneo completo de cada proyecto
  • Priorizar y resolver vulnerabilidades criticas/altas existentes
  • Configurar notificaciones Slack/email para nuevas vulnerabilidades
  • Documentar proceso de triaje semanal
Validacion
  • Verificar que PR con dependencia vulnerable es bloqueado
  • Verificar que alertas llegan correctamente
  • Test de bypass: intentar mergear sin resolver (debe fallar)
  • Verificar reportes semanales de estado
Dependencias

Acceso admin a repositorios GitHub. Puede comenzar en Semana 2.

Objetivo: Centralizar logs de todas las aplicaciones para deteccion de amenazas, usando Papertrail (integra con DO) o Datadog.

Aplicaciones Afectadas

Parse Hub Checkpoint Rendiciones AxiomaDocs Digital Ocean

Integracion Apps

Configurar Winston/Pino en cada app, structured logging

8 dias
Alertas y Dashboards

Crear alertas de seguridad y dashboard operativo

4 dias
Tareas de Implementacion
  • Evaluacion: Prueba de concepto con Papertrail (14 dias gratis)
  • Parse: Reemplazar console.log por Winston con formato JSON
  • Parse: Agregar logs estructurados: login attempts, failed auth, role changes
  • Hub: Configurar Winston, logs de integraciones
  • Checkpoint: Logs de fichadas, accesos
  • Rendiciones: Logs de transacciones financieras
  • AxiomaDocs: Logs de acceso a documentos sensibles

  • Alertas: Crear alerta: acceso admin fuera de horario laboral
  • Alertas: Crear alerta: error rate >5% en cualquier app
  • Dashboard: Panel con metricas: requests/min, errors, auth events
  • Retention: Configurar 30 dias de retencion (requisito seguro)
Validacion
  • Verificar que logs de todas las apps llegan al SIEM
  • Simular 10 failed logins y verificar alerta
  • Verificar busqueda: encontrar evento especifico por timestamp
  • Test de latencia: tiempo desde evento hasta visible en dashboard
  • Verificar que no se loguean datos sensibles (passwords, tokens)

5. Pentest Anual Esfuerzo:

Objetivo: Contratar pentest profesional para identificar vulnerabilidades no detectadas por herramientas automaticas.

Alcance

Parse (API + Frontend) Hub Infraestructura DO
RFP y Seleccion

Solicitar propuestas, evaluar 3 proveedores

5 dias
Preparacion

Ambiente de staging, documentacion, accesos

3 dias
Ejecucion

Pentest activo por proveedor externo

10 dias
Remediacion

Corregir hallazgos criticos y altos

10 dias
Tareas
  • Definir alcance: Web App Pentest + API Security + Infra basica
  • Solicitar propuestas a 3 proveedores (ej: Securetia, Base4, Faraday)
  • Evaluar propuestas: metodologia, experiencia, precio, certificaciones
  • Seleccionar proveedor y firmar NDA + contrato
  • Preparar ambiente de staging identico a produccion
  • Crear cuentas de prueba con diferentes roles
  • Documentar endpoints, flujos criticos, datos sensibles
  • Coordinar ventana de testing (evitar horarios criticos)
  • Acompanar ejecucion, responder consultas
  • Recibir reporte preliminar, revisar hallazgos
  • Priorizar remediacion: Criticos en 7 dias, Altos en 14 dias
  • Solicitar re-test de hallazgos corregidos
  • Obtener reporte final para evidencia de seguro
Entregables Esperados
  • Reporte ejecutivo (para gerencia/aseguradora)
  • Reporte tecnico detallado con PoC de vulnerabilidades
  • Matriz de hallazgos con CVSS score
  • Recomendaciones de remediacion
  • Carta de re-test confirmando correccion
Dependencias

Idealmente ejecutar despues de implementar MFA y SIEM (Semana 9+) para testear controles nuevos.

FASE 4: Mejoras de Prioridad Baja (Semanas 13-16)
9. Gestor de Secretos (Futuro) Esfuerzo:

Objetivo: Migrar secretos de variables de entorno a un gestor dedicado (HashiCorp Vault o DO Secrets) con rotacion automatica.

Aplicaciones Afectadas

Parse Hub Checkpoint Rendiciones AxiomaDocs
Evaluacion

Comparar Vault vs DO Secrets vs AWS SM

3 dias
Setup

Instalar y configurar gestor elegido

5 dias
Migracion

Migrar secretos de cada app, una por una

8 dias
Rotacion

Configurar politicas de rotacion automatica

4 dias
Tareas
  • Inventariar todos los secretos en .env de cada proyecto
  • Clasificar: DB credentials, API keys, JWT secrets, etc.
  • Evaluar opciones: Vault (self-hosted), DO Secrets, Doppler
  • PoC con la opcion elegida
  • Configurar acceso por aplicacion (cada app solo ve sus secretos)
  • Modificar codigo: leer secretos desde API del gestor
  • Parse: migrar primero (es el core)
  • Hub, Checkpoint, Rendiciones, Docs: migrar en orden
  • Configurar rotacion automatica para DB passwords (cada 90 dias)
  • Configurar alertas de expiracion de secretos
  • Eliminar .env de produccion una vez migrado
  • Documentar proceso de agregar nuevos secretos
Validacion
  • Verificar que apps inician correctamente leyendo secretos del gestor
  • Test de rotacion: rotar un secreto y verificar que app sigue funcionando
  • Test de acceso: app A no puede leer secretos de app B
  • Test de auditoria: verificar logs de acceso a secretos
Dependencias

Recomendado ejecutar despues de que las demas mejoras esten estables. No es urgente dado que los .env actuales son de testing.
